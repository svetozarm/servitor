# SERVITOR — Software As A Repository

A single binary that serves a static frontend, proxies API calls to a backend via CGI-style stdin/stdout, and auto-syncs data to git. Everything lives in one repo — frontend, backend, and data.

## Quick Start

Download servitor binary from the release page, or build it yourself, and then:

```bash
# Scaffold a new project
servitor init

# Build your backend (any language, reads HTTP from stdin, writes HTTP to stdout)
cd backend && go build -o servitor-backend . && cd ..

# Run
servitor
# → http://127.0.0.1:8080
```

## How It Works

1. **Static files** — requests not matching the API prefix are served from `frontend/`
2. **CGI proxy** — requests matching the API prefix spawn your backend binary, pipe the raw HTTP request to stdin, and read the raw HTTP response from stdout
3. **Git sync** — file changes in `data/` are auto-committed and pushed after a configurable inactivity delay

## Project Structure

```
my-app/
├── .servitor.conf          # Configuration (YAML)
├── frontend/           # Static files (HTML, CSS, JS)
│   └── index.html
├── backend/            # Your backend binary source
│   └── servitor-backend
├── data/               # Persistent data (auto-synced to git)
└── .git/
```

## Configuration

Create `.servitor.conf` in your project root (generated by `servitor init`):

```yaml
server:
  port: 8080
  host: 127.0.0.1

frontend:
  path: ./frontend

backend:
  path: ./backend/servitor-backend
  api_prefix: /api
  timeout: 3s

sync:
  enabled: true
  inactivity_delay: 30s
  max_interval: 5m
```

| Field | Description | Default |
|-------|-------------|---------|
| `server.port` | Port to listen on | `8080` |
| `server.host` | Bind address | `127.0.0.1` |
| `frontend.path` | Directory for static files | `./frontend` |
| `backend.path` | Path to backend binary | `./backend/servitor-backend` |
| `backend.api_prefix` | URL prefix that routes to the backend | `/api` |
| `backend.timeout` | Max time for backend to respond | `3s` |
| `sync.enabled` | Enable git auto-sync | `true` |
| `sync.inactivity_delay` | Time after last file change to trigger sync | `30s` |
| `sync.max_interval` | Maximum time between syncs regardless of activity | `5m` |

## Multi-App Mode

A single-app project is just a special case of multi-app — internally, servitor always resolves the config into a list of apps. A single-app config produces one app mounted at `/`.

To host multiple apps from a single repo, use a top-level config with an `apps` list:

```yaml
server:
  port: 8080
  host: 127.0.0.1

apps:
  - name: notes
    path: ./apps/notes
  - name: wiki
    path: ./apps/wiki
```

Each app directory contains its own `.servitor.conf` with `frontend`, `backend`, and `sync` settings. Named apps are served at `/<name>/` (e.g., `/notes/`, `/wiki/`). The root `/` shows an index page linking to all apps.

```
my-repo/
├── .servitor.conf          # Top-level: server + apps list
├── apps/
│   ├── notes/
│   │   ├── .servitor.conf  # frontend/backend/sync config
│   │   ├── frontend/
│   │   ├── backend/
│   │   └── data/
│   └── wiki/
│       ├── .servitor.conf
│       ├── frontend/
│       ├── backend/
│       └── data/
└── .git/
```

Each app gets its own sync engine and CGI proxy. The single-app config format continues to work unchanged — it is treated as one root-mounted app internally.

## Backend Contract

Your backend binary receives a **raw HTTP request** on stdin and must write a **raw HTTP response** to stdout.

Example request on stdin:
```
POST /api/notes HTTP/1.1
Host: 127.0.0.1:8080
Content-Type: application/json
Content-Length: 27

{"title":"hello","body":"w"}
```

Example response on stdout:
```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 15

{"status":"ok"}
```

The backend is spawned **per request** — recompile it anytime for instant hot-reload.

### Error Responses

| Condition | HTTP Status |
|-----------|-------------|
| Backend crashes (non-zero exit) | 502 Bad Gateway |
| Backend times out | 504 Gateway Timeout |
| Backend binary not found | 502 Bad Gateway |

## Git Sync Behaviour

When `sync.enabled` is true, servitor watches the `data/` directory:

- After a file change, a timer starts for `inactivity_delay`
- If no further changes occur, servitor commits and pushes
- If changes keep happening, sync fires at `max_interval` regardless
- On conflict (diverged remote), servitor **force pushes** (local wins)
- On shutdown (SIGTERM/SIGINT), pending changes are flushed before exit

## CLI

```bash
servitor              # Start the server (reads .servitor.conf from current directory)
servitor init         # Scaffold a new project with default template
servitor init NAME    # Scaffold using a named template variant
servitor init --repo URL  # Use a custom template repository
```
